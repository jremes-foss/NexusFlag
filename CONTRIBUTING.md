# Contributing

NexusFlag is an Open Source, community-driven project. I would be very happy to receive help making this project better and contributions are very welcome. If you'd like to contribute, please read the following guidelines in this document.

When contributing to this repository, please first discuss the change you wish to make via issue,
email, or any other method with the owners of this repository before making a change.

## Conventions

Conventions
The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT" "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt).

## Code of Conduct

This repository is following [Code of Conduct](CODE_OF_CONDUCT.md). Please familiarise yourself with this document as you must follow these guidelines while interacting with the project.

## Creating Issues

One way you can make NexusFlag better project is by pointing out issues. This can be anything from pointing out typos to larger feature requests. You should put enough information to understand the issue properly. If you are reporting bugs, you should enter the following information:

- A description of the issue, and how to reproduce it.
- An example of location where this bug occurs. 

Especially the last one is very important to debug the issue and see where the bug occurs.

## Pull Requests

Pull requests to NexusFlag are welcome but they should follow the best practices.
- Subject line should be kept under 50 characters.
- Description should be kept informative. For example: why was change necessary and how does the pull request address the problem?
- If there is an issue open regarding the pull request, please link the open issue to description.

### Commit Messages
NexusFlag is following the 50/72 rule:
- First line of your commit message should be maximum 50 characters long.
- Each line in your description should wrap at the 72nd mark.

Please note that this rule is not strictly observed, it is more like "rule of thumb" guideline But regardless it should be treated as best practice, even if not followed in a strict manner.
